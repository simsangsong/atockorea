# ETL 멈춤/지연 가능 원인 분석

## 1. 입력 파일(visitkorea.nt) 관련

| 원인 | 분석 결과 | 비고 |
|------|-----------|------|
| **새줄 없는 거대 라인** | 전체 파일 스캔 결과 **최대 11,350바이트**(약 11KB) 구간만 존재. 2MB 구간 없음. | 2MB 건너뛰기 로직은 실제로 거의 걸리지 않음. |
| **줄끝 문자 (\r only)** | 일부 구간에서 `\n` 없이 `\r`만 있으면 청크 리더가 한 줄을 매우 길게 인식할 수 있음. | 40.5MB 근처 100KB 구간에서 `\n`/`\r` 0개 확인된 적 있음 → 해당 구간은 실제로 새줄 없음. |
| **파일 경로/권한** | `C:\Users\sangsong\Downloads\visitkorea.nt` 사용 중. | 다운로드 폴더는 백신 실시간 검사 대상일 수 있음. |
| **디스크 I/O 지연** | `f.read(256*1024)` 호출 시 디스크가 느리거나 슬립/절전이면 블로킹. | 네트워크 드라이브/외장 HDD/백신 스캔 시 심해짐. |

## 2. Generator(read_lines_bounded) 관련

| 원인 | 설명 | 가능성 |
|------|------|--------|
| **f.read(CHUNK) 블로킹** | 256KB 읽을 때 OS/디스크가 응답 지연하면 그동안 yield 안 함 → 진행 정지처럼 보임. | **높음** (특히 27만·28만 줄 근처 특정 오프셋에서 반복 시). |
| **buf.encode('utf-8') 비용** | `len(buf.encode("utf-8")) > MAX_LINE_BYTES` 체크 시 buf가 크면 인코딩 비용 발생. | buf는 보통 256KB 이하이므로 중간. |
| **텍스트 모드 디코딩** | `open(..., encoding='utf-8', errors='replace')`로 256KB 읽을 때마다 디코딩. | 대용량 파일에서 누적되면 지연 가능. |

---

## 3. 메인 루프(파싱/기록) 관련

| 원인 | 설명 | 가능성 |
|------|------|--------|
| **정규식 ReDoS** | `RE_NT_LINE`의 `"[^"]*(?:\\.[^"]*)*"` 등이 특정 패턴에서 catastrophic backtracking. | **중간** (긴 줄·특수 문자 조합에서만). 50,000자 초과 줄은 이미 스킵함. |
| **parse_nt_line 호출 비용** | 줄마다 정규식 match + 문자열 처리. | 정상 줄만 있으면 보통 낮음. |
| **records dict 성장** | 4만~5만 개 리소스 시 dict 조회/삽입. | Python dict는 O(1)이라 보통 문제 안 됨. |
| **write_progress 비용** | 1,000줄마다 파일 열기·쓰기·flush·fsync. | fsync 시 디스크 대기로 소폭 지연 가능. |

## 4. 실행 환경 관련

| 원인 | 설명 | 가능성 |
|------|------|--------|
| **stdout 블로킹** | 백그라운드 실행 시 터미널이 stdout을 읽지 않으면, print()가 파이프 버퍼 가득 찼을 때 블로킹. | 읽기 구간에서는 print 제거해 완화함. |
| **Windows 백신/Defender** | visitkorea.nt·places.db·etl_progress.txt 접근 시 실시간 검사로 I/O 지연. | **높음** (특히 27만 줄 근처 같은 특정 오프셋 반복 스캔). |
| **디스크 슬립/절전** | HDD/외장 디스크가 일정 시간 후 슬립하면 다음 read에서 깨울 때까지 대기. | 가능. |
| **Cursor/IDE 터미널** | 백그라운드 프로세스의 stdout/스레드 스케줄링이 제한적일 수 있음. | 가능. |

## 5. 진행 파일(etl_progress.txt) 관련

| 원인 | 설명 | 조치 |
|------|------|------|
| **파일 버퍼링** | 쓰기 후 OS/파일시스템이 버퍼만 채우고 디스크 반영 지연. | `flush()` + `fsync()` 적용함. |
| **에디터 캐시** | Cursor/VS Code가 열어둔 파일을 디스크 변경으로 자동 재로드하지 않음. | 수동 재열기 또는 `Get-Content -Wait`로 확인. |
| **첫 갱신 시점** | `write_progress(0,0,"started")` 후 다음 기록이 **1,000줄** 뒤. | 처음 몇 초는 lines=0으로만 보일 수 있음. |

## 6. SQLite 관련

| 항목 | 설명 |
|------|------|
| **읽기 단계** | 파싱 루프에서는 `conn`에 쓰지 않고 `records` dict만 갱신. DB I/O는 파싱 완료 후 일괄 삽입 단계에서만 발생. |
| **메모리** | records가 5만 건 수준이면 수 MB~수십 MB로 일반적으로 문제되지 않음. |

## 7. 요약: 가능성 높은 원인 순

1. **디스크/백신 I/O**  
   - `f.read(256*1024)` 또는 `write_progress` 시 디스크·백신이 특정 오프셋/파일에서 지연.  
   - 27만·28만 줄 근처 **바이트 오프셋**이 반복되면 그 구간에서만 멈춘 것처럼 보일 수 있음.

2. **stdout 블로킹(과거)**  
   - 1만 줄마다 print 하던 구간에서 파이프가 차면 블로킹.  
   - 현재는 읽기 루프에서 print 제거해 완화됨.

3. **정규식 ReDoS**  
   - 특정 길이/패턴의 줄에서 `RE_NT_LINE.match()`가 극단적으로 오래 걸릴 수 있음.  
   - 50,000자 초과 줄 스킵으로 일부만 완화됨.

4. **진행 파일 갱신 인지**  
   - 1,000줄마다만 갱신 + 에디터가 자동 재로드 안 함 → “멈춘 것”처럼 보일 수 있음.

## 8. 멈춤 구간 공통점 (276k vs 303k)

| 구간   | 줄 유형 예시 |
|--------|----------------|
| 276000 / 276600 | 276600은 **`owl#inverseOf`** (skos#broader/narrower 등). 276609·610은 **`rdf-schema#comment`+`\u`**. → **패턴 스킵**: `owl#sameAs`·`owl#inverseOf` 및 `rdf-schema#comment`+`\u` 포함 줄 파싱 전 건너뜀 |
| 303000 | `rdf-schema#label` + 객체 `"..."@ko` 한글 리터럴 (125~174자) |

- 공통점: 둘 다 정규식 `RE_NT_LINE`에서 객체 부분 `"[^"]*(?:\\.[^"]*)*"` 등이 특정 패턴에서 ReDoS/극단적 백트래킹을 유발할 수 있는 형태.
- 대응: **줄 단위 파싱 타임아웃**(기본 2초). 해당 줄만 스킵하고 `etl_skipped_lines.log`에 줄 번호 기록 → 반복 구간에서도 “문제 줄”만 최소 단위로 건너뜀.

### 8.1 타임아웃으로 스킵된 4줄 패턴 (297427, 298582, 301893, 323313)

| 항목 | 공통점 |
|------|--------|
| 서술어 | 모두 `rdf-schema#label` |
| 객체 | 따옴표 리터럴 안에 **백슬래시 이스케이프 유니코드** 다수 (`\uAD6C`, `\uB9AC` 등) |
| 길이 | 251~264자 |
| 내용 예 | `"[구리 루레이길 3코스] ..."`, `"만연사 선캄브리아기 ..."` 한글 라벨 |

- **ReDoS 원인**: 정규식 객체 부분 `"[^"]*(?:\\.[^"]*)*"` 에서, 리터럴 안 `\uXXXX` 반복 시 `\`+한글자 매칭이 조합 폭발 → 극단적 백트래킹.
- **결론**: `rdf-schema#label` 이면서 리터럴에 `\u` 가 많이 들어간 줄만 2초 초과로 스킵됨. 해당 패턴은 4줄뿐이라 손실 미미.

### 8.2 325540 등에서 "스킵도 안 됨"으로 멈추는 이유

- **파싱 타임아웃**은 **한 줄을 파싱할 때**만 적용됨.
- 멈춤이 **파일 읽기** 쪽이면, 다음 줄을 받지 못해 파싱 단계에 안 들어감 → 타임아웃이 걸리지 않음.
- **대응**: (1) **읽기 타임아웃** 추가: 60초 동안 새 줄이 없으면 "읽기 블로킹"으로 간주하고 종료 후 `SKIP_PARSE_RANGES` 안내. (2) 청크 16KB로 축소해 I/O 대기 구간 단축.

### 8.3 구간 325540~325550, 329080~329090 분석 (실제 추출 결과)

| 구간 | 줄 번호 | 특징 |
|------|---------|------|
| **325540~325550** | 11줄 | 모두 `rdf-schema#label` + 객체 `"..."@ko`, 리터럴 내 **`\\uXXXX` 다수** (146~320 bytes) |
| **329080~329090** | 11줄 | 동일 패턴 (119~348 bytes) |

- **공통**: 객체가 `"\\uB354\\uBCA0\\uC774..."`, `"\\uACBD\\uAE30\\uC61B\\uAE38..."` 형태로 **백슬래시 이스케이프 유니코드**가 연속된 한글 라벨. 8.1에서 정리한 ReDoS 유발 패턴과 동일.
- **길이**: 119~348자이므로 현재 ETL의 `use_safe = len(line) > 2_000` 조건에 걸리지 않음 → **정규식 `parse_nt_line` 사용** → 해당 구간에서 `RE_NT_LINE.match()` 백트래킹으로 2초 내외 또는 그 이상 소요 가능.
- **주의**: 325541(320 bytes), 329082(348 bytes)처럼 `\uXXXX`가 특히 많은 줄에서 지연/타임아웃 스킵이 발생할 수 있음.
- **구현**: 해당 패턴이 나오면 **파싱 시도 없이** 해당 줄만 스킵. 조건: `rdf-schema#label` 이면서 리터럴에 `\u` 포함 시 → `parse_nt_line` 호출 전에 `continue` 하고 `etl_skipped_lines.log`에 `line=... # reodos_label_unicode` 기록. 동일 패턴이 어디서든 다시 나와도 걸리지 않고 건너뜀.

## 9. 권장 대응

- **실행 시**
  - `python etl_process.py > etl_log.txt 2>&1` 로 stdout 리다이렉트 유지.
  - 가능하면 visitkorea.nt를 백신 제외 경로(예: 프로젝트 폴더)로 복사 후 그 경로를 INPUT_NT로 사용.
- **진행 확인**
  - 터미널에서 `Get-Content etl_progress.txt -Wait` 로 실시간 확인.
- **추가 완화(선택)**
  - 진행 기록을 1,000줄마다가 아니라 100줄마다 하도록 줄여 “멈춤” 구간을 더 잘 구간 짓기.
  - `read_lines_bounded`에서 `buf.encode('utf-8')` 대신 `len(buf)`만 사용(대략 비교용)해 인코딩 비용 줄이기.
  - 정규식 대신 단순 split/인덱싱으로 파싱하는 “안전 모드” 경로 추가(긴 줄만 해당 경로 사용).
