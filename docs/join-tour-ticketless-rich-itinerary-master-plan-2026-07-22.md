### B-2. 기사 도착 플로우 = 3요소만
출발시간+주차핀(✅ A0 번들 그대로) + **출발 전 인원체크**(🔶 GuideConsole 탑승 X/Y는 있으나 **콕핏에 없음**; pure driver는 rooms[0] 단일룸 제약도 있음) → J2.

### B-3. 오프라인 안내 + 콘텐츠 암호화(加密)
- 현재: `OfflineInfoCard` localStorage **텍스트 스냅샷**(집합·일정·긴급번호)만.
- 목표: 도착 번들·스톱 콘텐츠(+이미지, 동영상은 결정 후)를 **사전 다운로드**해 오프라인에서 안내가 계속 뜨게. 단 **다운로드 파일은 암호화**해 원본 유출·경쟁사 도용 방지.
- 설계(J3): IndexedDB에 AES-GCM 암호문 저장, 키는 룸 세션에서 파생(WebCrypto, 키 자체는 저장 안 함 → 세션 없는 자는 캐시 파일을 열어도 암호문뿐). 서비스워커는 암호문 blob만 캐시.
- **정직한 한계(기록)**: 클라이언트 암호화는 DRM이 아님 — 실행 중 화면·메모리에서는 평문. 방어 목표는 "파일 시스템/캐시에서 원본 JSON·미디어를 통째로 퍼가는 것"의 차단이며, 그 수준은 달성 가능.

### B-4. 양방향 소통·관제 상시
✅ 이미 충족(무제한 번역 채팅·SOS 기사+관제 동시·컨시어지 에스컬레이션·손님측 오프라인 송신큐 T0-4). J3에서 오프라인 시 수신 캐시와 재연결 동기화만 재검증.
