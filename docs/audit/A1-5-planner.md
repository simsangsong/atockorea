## 1. 🔴 P1 — 가이드가 확정한 뒤에도 에디터가 "편집기인 척"했다

- **재현 경로:** 손님이 폰에서 `/tour-mode/plan/[bookingId]`를 열어 스톱을 고르는 중,
  가이드가 콘솔에서 [확정]을 누른다(`GuidePlanPanel`). 손님은 계속 편집한다.
- **관측된 동작:** 자동저장 PUT이 서버에서 **409 `plan_locked`** 로 거절된다
  (`plan/route.ts:436` — 확정된 일정은 손님이 못 고친다. 서버는 옳다).
  클라이언트는 409를 **다른 모든 실패와 똑같이** 취급했다(`saveState = 'error'`) →
  화면 하단에 "저장하지 못했어요" 한 줄만 뜨고, **탭·[담기]·순서 변경·삭제가 전부 그대로 살아 있다.**
  손님은 계속 편집하고, 편집한 것은 **하나도 저장되지 않는다.**
- **기대 동작:** 저장할 수 없는 화면은 편집기처럼 보이면 안 된다.
- **왜 P1인가:** "저장 실패"는 **다시 해보라**는 뜻으로 읽힌다. 실제로는 영원히 실패한다.
  손님은 자기가 방금 10분간 짠 일정이 사라졌다는 것을 **투어 당일에야** 안다.
  게다가 이 화면에는 이미 정확히 이 상황을 위한 문구가 **5로케일 전부** 준비돼 있었다 —
  `confirmedNote`("가이드가 일정을 확정했어요. 변경하고 싶은 게 있으면 투어룸 채팅으로 알려주세요").
  **문구는 있는데 그 문구로 가는 길이 없었다.**
- **조치:** 409를 받으면 `/plan`을 **한 번 다시 읽는다**(`resyncLockedPlan`).
  서버의 진짜 상태를 채택하면 `isConfirmed`/`isSubmitted`가 켜지면서 편집 표면이
  통째로 읽기 전용으로 바뀌고 `confirmedNote`가 뜬다. **새 문구·새 상태값 없이** 기존 배너를 재사용한다.
  비용은 GET 1회, 그것도 **실제 충돌일 때만**.
  다른 기기에서 이미 제출(`guest_submitted`)한 경우도 같은 경로로 정리된다.
- 회귀: `__tests__/components/tour-mode/planEditorLock.test.tsx`
  (수정 전 상태로 되돌려 **실패하는 것을 확인**한 뒤 복구).

## 2. 🟡→✅ P2 — 저장이 끝난 초안을 탭 전환마다 다시 보냈다

- **재현 경로:** 스톱을 하나 담는다(자동저장 2.5초 뒤 완료). 지도를 보러 앱을 잠깐 벗어난다. 돌아온다. 반복.
- **관측된 동작:** `scheduleAutosave`가 디바운스 타이머 핸들(`saveTimer.current`)을
  **발사 후에도 비우지 않았다.** `flushPending`은 "핸들이 남아 있으면 저장할 게 남았다"로 판단하므로,
  **첫 편집 이후 세션 내내** 백그라운드 전환·언마운트마다 이미 저장된 초안을 `keepalive` PUT으로 다시 보냈다.
  서버는 그때마다 좌표 보강 + 실행가능성 재계산 + 행 업서트를 한다(공짜가 아니다).
- **왜 고쳤나:** 손님을 기다리게 하지 않는 낭비지만, **§L-D1의 반대 방향으로 정확히 부합한다** —
  성능을 깎지 않고 제거되는 순수 낭비. 수정은 1줄(`saveTimer.current = null`)이고,
  `submitPlan`/`delegatePlan`의 `clearTimeout`에도 같은 누락이 있어 함께 정리했다.
- **경계:** "아직 저장 안 된 초안은 여전히 플러시된다"를 테스트로 같이 고정했다 —
  이 수정이 원래 의도(빠른 이탈 시 최대 2.5초 유실 방지)를 깨지 않았다는 증거가 필요하다.

## 3. 🟡→✅ P2 — 깨진 사진이 카드 최대 요소에서 브로큰 아이콘으로 남았다 (`PlanStopCards`)

- **관측된 동작:** 스톱 카드의 92×92 썸네일이 `<img>` 그대로였다. 자산 URL이 죽으면
  브라우저 기본 깨진-이미지 글리프가 카드에서 **가장 큰 요소** 자리에 남는다.
- 🔴 **같은 파일 안에 이미 정답이 있었다.** `PlanEditorClient`의 POI 피커는
  `onError`로 스와치로 강등하고 주석까지 달려 있다 —
  *"Dead asset URLs degrade to the plain swatch, not a broken-image glyph."*
  피커 썸네일은 40px대인데 **그쪽은 처리하고, 훨씬 큰 이 카드는 안 하고 있었다.**
- **조치:** 깨진 URL을 상태에 기억해 이미 있는 `Sparkles` 스와치 분기로 떨어뜨린다(5줄).

## 5. 🟡 P2 백로그 — 코스 미리보기 시트가 `Sheet`를 안 쓴다

`PlanEditorClient:2338`의 미리보기 오버레이는 `role="dialog"` + `aria-modal`과 로케일 라벨은 있지만
**Esc 처리·포커스 이동/복원·배경 스크롤 잠금이 없다.** `Sheet.tsx`가 그 전부를 갖고 있다(A1.3 기록).
**즉시 고치지 않은 이유:** 2,460줄 화면의 마크업을 재배치해야 하고, 이 표면은 모바일 전용이라
Esc가 사실상 도달하지 않으며 배경 탭 닫기는 동작한다 — 위험 대비 이득이 낮다. A8.2 백로그.

## 6. 🔵 P3 — 헤더 눈썹 문구 `Smart guide planner`가 영어 고정

## 7. 확인했고 문제없음

1. **5로케일 카피가 타입으로 강제된다** — `Record<RoomLocale, PlanCopy>`라 키 하나만 빠져도 `tsc`가 운다.
   A0.4가 지적한 "런타임에야 드러나는 로케일 구멍"이 이 화면엔 구조적으로 없다.
2. **실행가능성 경고가 코드 기반이다**(`overrun`/`closed`/`out_of_region`/`cross_island`).
   서버가 문장을 만들어 보내지 않고 **코드만** 보낸다 → 손님 로케일로 렌더된다. 서버 영어 누출 없음.
3. **Places 검색이 경합에 안전하다** — 효과별 `cancelled` 플래그로 늦게 온 응답이 새 결과를 덮지 않는다.
   300ms 디바운스 + 2글자 미만 미조회로 호출도 아낀다.
4. **순서 변경이 위/아래 버튼이다**(드래그 전용 아님) + `aria-label`이 로케일 문구.
   모바일·스크린리더 양쪽에서 도달 가능하다.
5. **중복 제출 차단이 ref로 되어 있고**(`submittingRef`) 서버도 `guest_submitted` 재제출을 200으로 흡수한다 —
   피드 캡슐이 두 번 뜨지 않는다. 클라이언트·서버 **양쪽**에 방어가 있다.
6. **빈 일정 제출 불가**가 버튼 비활성(`stops.length === 0`)과 서버 400 양쪽에 있다.
7. **`PlanTourItinerary`는 스톱이 없으면 렌더하지 않는다** — 빈 섹션 껍데기를 남기지 않는다.
