**대상:** `ConciergePanel` `ConciergeInlineAnswer` `ConciergeEntryRow` `Sheet` (4개, 547줄)

**판정:** P0 없음 · P2 1건. 단 그 1건이 **플랜의 측정 방법 하나를 무효화**한다 — §2 참조.

## 1. 🔴 확인했고 문제없음 — `Sheet.tsx`는 이 저장소의 모범이다

A1.1이 `Lightbox`에 없다고 지적한 계약이 **여기엔 전부 있다.** 대조 대상으로 기록해 둔다.

| 계약 | `Sheet` | `Lightbox` |
|---|---|---|
| `onClose` ref화 (인라인 함수로 인한 이펙트 재실행 차단) | ✅ `onCloseRef` | ❌ 의존성에 직접 |
| `role="dialog"` + `aria-modal="true"` | ✅ | ❌ |
| 열 때 패널 포커스 · 닫을 때 **원래 요소로 복원** | ✅ `restoreRef` | ❌ |
| 배경 라벨 로케일화 | ✅ `closeLabel` prop | ❌ 영어 고정 |
| reduced-motion 존중 | ✅ | — |

그리고 **왜 ref가 필요했는지가 코드 주석에 사고 재현 경로로 남아 있다**:

> *tapping the concierge input opened the keyboard → visualViewport resize →
> RoomShell re-render (useKeyboardOpen) → focus stolen → keyboard closed.
> Guests could never type into the Smart Guide.*

결정론적 잠금이었다. 이 주석은 지우지 말 것 — A1.1이 `Lightbox`에서 같은 패턴을 찾아낸 것도
이 기록 덕분이다.

## 2. findings

### 🟡 P2 — Tier 0 답변이 **어디에도 기록되지 않는다** → A7.4의 측정 방법이 성립하지 않는다

- **재현 경로:** `ConciergePanel.tsx:131-142`(칩 탭)와 `:145-175`(자유 입력).
  손님이 "화장실 어디예요"를 묻고, 서버 로그(`chat_messages`)를 확인한다.
- **관측된 동작:** `fetch`는 **단 한 곳**(`askServer`)에서만 일어나고, 거기로 가는 경로는
  **`restaurant`와 `ops_request` 둘뿐**이다. 나머지는 전부 클라이언트에서 끝난다:
  - 칩 4종(화장실·포토스팟·다음 스톱·남은 시간) → `answerTier0` 로컬 응답
  - 자유 입력의 Tier 0 의도 적중 → 로컬 응답
  - `emergency` 가드레일 → 로컬 응답
  - `venue_recommendation` 가드레일 → 로컬 거절
  `ConciergeInlineAnswer`는 아예 *"never persisted or broadcast"*라고 명시한다.
  즉 **`chat_messages`에는 Tier 1로 샌 질문만 남는다.**
- **기대 동작:** 무엇을 기대할지가 이 finding의 핵심이다 — 아래 참조.

#### 🔴 왜 이게 단순 로깅 누락이 아닌가

플랜 **A7.4**의 정의: *"실제 손님 질문 로그(`chat_messages`) 대비 Tier0 적중률"*.

그런데 **Tier 0 적중은 그 테이블에 들어가지 않는다.** 적중률의 분자가 구조적으로 0이고,
분모는 "적중하지 못한 질문"만 담는다. 이 정의대로 재면 **Tier0 적중률은 항상 0%로 나온다** —
사전이 완벽하게 동작해도 그렇다. 측정이 틀린 답을 주는 것이 아니라, **측정이 불가능하다.**

같은 이유로 §A **S1 선제성**의 "되물음률 < 10%"도 지금은 잴 수 없다. 첫 질문과 되물음이
둘 다 로컬 응답이면 서버는 어느 쪽도 못 본다.

⚠ **§L L4는 영향받지 않는다.** L4가 세는 것은 "Tier 1로 **샌** 질문"이고 그건 서버에 도달한다.
L4 스크립트의 출력은 지금도 정확하다.

#### 왜 "전부 로깅"이 정답이 아닌가

로컬 응답을 서버에 기록하려면 왕복이 생긴다. 그건 Tier 0의 존재 이유(§F "네트워크 0회",
p50 1.0µs — A0.3 실측)와 **§L-D1**("절감이 손님을 기다리게 하면 채택하지 않는다")을 동시에 어긴다.
그래서 선택지는 셋이고, 판단은 A0.2/A7.4가 해야 한다:

1. **정의를 고친다** — A7.4를 "Tier1 유출량"으로 재정의(§L L4가 이미 그걸 잰다). 비용 0.
2. **비동기 카운터** — 로컬 응답 시 `sendBeacon`으로 **의도 코드만**(질문 원문 없이) 묶어 전송.
   손님을 기다리게 하지 않고, PII도 안 실린다.
3. **그대로 둔다** — 대신 A7.4/S1을 "측정 불가"로 명시. 🔴 지금처럼 **측정 가능한 척** 두는 것만은
   피해야 한다(플랜이 다이닝 MISS 8초에서 이미 겪은 부류).

**이 티켓에서 고치지 않는 이유:** 셋 중 어느 것이든 A0.2(관측 도구)의 설계 결정이고,
컨시어지 컴포넌트만 고쳐서 될 일이 아니다. A0.2 착수 시 이 문서를 읽을 것.

## 3. 그 밖에 확인했고 문제없음

1. **가드레일이 클라이언트·서버 같은 모듈을 쓴다** (`lib/tour-room/concierge`의
   `classifyConciergeGuardrail`). 판정이 두 벌이 아니라 드리프트가 없다 — A4.1이 잡는 부류의 반대 사례.
2. **`ops_request`는 반드시 서버로 간다.** 로컬에서 끝내면 에스컬레이션 메시지가 피드에
   안 올라간다는 것을 주석이 명시한다(*"ops requests must reach the server"*). 옳다.
3. **음식 질문이 가드레일보다 먼저 판정된다** — 데이터 기반 답이 "업체 추천 거절"보다 우선한다
   (`submit()`의 `intentEarly` 검사가 `classifyConciergeGuardrail`보다 위). 순서가 의도적이고
   주석에 이유가 있다. 뒤집으면 다이닝 RAG가 통째로 죽는다.
4. **`provider`·`cached`가 손님 화면에 노출되지 않는다.** 패널은 `text`/`mapCard`/`card`만 읽는다 —
   §L L2가 응답에 추가한 `cached: true`는 내부 신호로만 남는다.
5. **`ConciergeInlineAnswer`가 피드를 오염시키지 않는다.** 질문자 화면에만 뜨고 저장·브로드캐스트
   안 한다 — 공용 피드를 사람 채널로 유지하고 가이드의 답을 덮어쓰지 않는다는 설계가 명시적이다.
6. **에러 문구가 로케일 상수다** (`copy.error`). 서버가 죽어도 손님은 자기 언어로 실패를 본다.
