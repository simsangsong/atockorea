# A0-sim-env — 시뮬 환경 격리 (A0.1)

**대상:** `scripts/sim-tour-day.ts` · `lib/ops/sim/simScope.ts` · 집계·금전 표면
**감사일:** 2026-07-24 · **축:** 2 기능 · 3 코드 · 7 안전
**판정:** ✅ **통과** — 실 예약 오염 0으로 시뮬 가능. 시더는 가드 없이 실행 불가. cleanup 후 고아 행 0.

---

## 0. 무엇이 문제였나

`sim-tour-day.ts`가 라이브 `bookings`에 테스트 예약을 직접 넣는다. 오너가 2026-07-23에
테스트 예약을 전부 지우고 실주문 2건만 남긴 상태(D8)인데, 시뮬을 돌릴 때마다 그게 되살아난다.

**그 결과 지금까지 일어난 일:** §11.A·§11.G의 UI 검증이 "스크린샷은 의도적 생략 — sim 시딩이
프로덕션에 테스트 예약을 재주입"으로 **반복 연기**됐다(§14.6 G9). A5(실사용 시뮬)와 A2(운영자 3면)가
통째로 막혀 있었고, B1(통합 통계)은 검증할 데이터 자체가 없었다.

## 1. 대안 검토와 채택

| 안 | 판정 | 이유 |
|---|---|---|
| 로컬 Supabase (`supabase start`) | ❌ | Windows Docker 의존 + 마이그레이션 25개 재현 + env 스위칭. 하루 안에 되는 작업이 아니다 |
| Supabase 브랜치 | ❌ | 과금. 오너 승인 없이 쓸 수 없다(D10 정신) |
| 먼 미래 날짜로 격리 | ❌ | 앱이 `kstToday()`로 "오늘의 투어"를 잡는다. 투어 당일 시나리오가 **성립하지 않는다** — 시뮬의 목적 자체가 사라진다 |
| **행 태깅 (`bookings.sim_tag`)** | ✅ **채택** | 시뮬 행이 라이브에 살되, **집계와 돈이 그것을 세지 않는다** |

## 2. 핵심 설계 결정

### 🔴 D-A0.1-1 — "숨기기"가 아니라 "세지 않기"다

시뮬 예약은 **룸·좌석·체크인·명단에서 정상적으로 보여야 한다.** 그게 시뮬의 목적이다.
전역 필터로 숨기면 시뮬이 시뮬 노릇을 못 한다.

배제 대상은 딱 두 종류:

| 종류 | 표면 | 왜 |
|---|---|---|
| **집계** | §11.E 일일보고서 · §K B1 통합통계 · §K B2 정원 카운트 | 오너가 매일 아침 존재하지 않는 투어를 본다 |
| **돈** | Stripe 캡처 크론 · `ops_entity_ledger` 법인 원장 | §6 3자 대사가 깨진다 |

### 🔴 D-A0.1-2 — 30개 호출부 전부에 필터를 달지 않는다

`from('bookings')` 호출부는 라이브 코드에 **29곳**이다. 전부에 필터를 다는 설계는
드리프트를 보장한다(§H-4가 이 저장소의 반복 실패모드로 지목한 것 — 새 화면 하나가 필터를
빠뜨리면 조용히 오염된다). **해가 되는 표면만** 고르고, 그 목록을 `simScope.ts` 주석에 적었다.
새 집계 화면을 만들 때 그 파일을 읽게 하는 것이 목적이다.

### 🔴 D-A0.1-3 — 예약만 빼면 "예약 없는 룸"이 생긴다

집계 대부분이 `tour_rooms`에서 출발해 `.in('id', bookingIds)`로 예약을 되짚는다.
예약 쪽에만 쿼리 필터를 걸면 **룸은 남고 예약만 사라져** 존재하지 않는 상태가 집계에 들어간다.
그래서 `dropSimBookings`는 결과 필터이고, `simBookingIds`로 **그 예약의 룸까지 함께** 뺀다.
예약에서 **직접** 출발하는 집계(신규 예약 목록·캡처 대상)만 쿼리 레벨 `.is('sim_tag', null)`을 쓴다.

### 🔴 D-A0.1-4 — 원장 가드는 fail-open이다

`recordCaptureLedger`가 쓰기 전에 `sim_tag`를 확인한다. 단 **조회가 실패하면 진행한다.**
1차 방어는 캡처 크론의 쿼리 배제이고 이건 심층 방어다. 여기서 fail-closed로 막으면
일시적 DB 오류가 **실제 매출을 원장에서 조용히 누락**시킨다 — 시뮬 유입보다 나쁘고,
아무도 눈치채지 못한다.

## 3. cleanup이 실제로 깨져 있었다 (신규 발견)

기존 `--cleanup`은 `bookings`·`tour_rooms`·메시지·참가자·위치·초대만 지웠다.
라이브 FK를 실측한 결과 **대부분은 `ON DELETE CASCADE`라 예약만 지워도 따라 사라진다**
(좌석·차량·이벤트·엑스트라·핀·노쇼증거·푸시구독·일정).

문제는 **`ON DELETE SET NULL`인 것들**이다 — 예약을 지우면 **고아 행이 라이브에 영구 잔존**한다:

| 테이블 | 컬럼 | 규칙 | 위험 |
|---|---|---|---|
| **`ops_entity_ledger`** | `booking_id` | SET NULL | 🔴 **법인 원장에 시뮬 매출이 남고, 지운 뒤엔 어느 행이 시뮬이었는지 알 방법조차 없다** |
| `ops_email_parse_logs` | `booking_id` | SET NULL | 파싱 통계 오염 |
| `ops_guide_assignments` | `booking_id` | SET NULL | 가이드 정산 대상 오염 |
| `emails` · `reviews` · `promo_code_usage` · `received_emails` · `coupon_grants` | — | SET NULL | 잔재 |
| `coupon_redemptions` | `booking_id` | **NO ACTION** | 🔴 삭제 자체를 **차단**한다 |

**수정:** cleanup이 예약을 지우기 **전에**, `booking_id`가 아직 시뮬을 가리키는 동안
위 9개를 먼저 지운다. 순서가 전부다.

## 5. 적용 후 검증 (라이브)

```
col=1 · real_bookings=3 · sim_bookings=0 · idx=1
```
실주문 3건 전부 `sim_tag IS NULL`. 소급 태깅 대상 0건(과거 시뮬 잔재는 07-23에 이미 정리됨).

## 6. 확인했고 문제없음

1. **`scripts/.sim-fixtures.json`은 이미 `.gitignore` 83번 줄에 있다** (A-plan-review R13). 서명 토큰 유출 경로 없음.
   그래도 cleanup이 파일을 지우도록 추가했다 — 만료 전 토큰이 디스크에 남을 이유가 없다.
2. **캡처 크론은 원래도 시뮬을 잡지 못했다.** 시더가 `payment_method`를 안 채우고 쿼리가
   `.eq('payment_method','stripe')`이기 때문. 다만 **우연에 기댄 안전**이라 명시 배제를 넣었다.
3. **Stripe 웹훅 경로**는 실제 `payment_intent.succeeded`에서만 발화한다. 시뮬 예약엔 payment intent가
   없으므로 발화 불가. 원장 가드가 그래도 뒤를 받친다.
4. 기존 `bookings` 소비처 중 **per-booking 조회**(링크 발급·명단·체크인 컨텍스트·좌석)는 배제하지 **않았다** —
   시뮬이 동작해야 하는 바로 그 경로다.

## 7. 남은 것 (이 티켓 밖)

- **B1 통합 통계**는 착수 시점에 처음부터 `simScope`를 쓴다(D-A0.1-2의 "새 집계 화면" 케이스).
- **B2 정원 카운트**도 동일.
- 시뮬을 실제로 1회 주행해 스크린샷을 남기는 것은 **A5**의 일이다. A0.1은 그 문을 연 것까지다.
