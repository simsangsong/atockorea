# A0-perf-baseline — 성능 베이스라인 (A0.3)

**대상:** §F 성능 예산 10개 지표 · **감사일:** 2026-07-25 · **축:** 5 속도
**기준선:** main `fbe95838` · node v24.14.0
**재실행:** `npx tsx scripts/perf-baseline.ts` (번들 수치는 `next build` 선행 필요)

> **판정: 부분 통과.** 지금 **정직하게 잴 수 있는 4개**는 전부 실측해 예산 대비로 기록했다.
> 나머지 6개는 서버·실기기가 있어야 하며, **추정치를 적지 않는다.**

---

## 🔴 왜 추정치를 안 쓰는가

플랜 자신이 이미 값을 치렀다. §F의 다이닝 MISS 예산 "8초"는 **근거 없는 추정**이었고,
실측하니 틀렸다 — 플랜 본문이 그 자리에 이렇게 적어 뒀다:

> *"8s는 근거 없는 추정이었다. … 예산은 측정 후에 쓴다"*

베이스라인에 추정이 한 줄이라도 섞이면 **다음 세션이 그 숫자를 기준으로 회귀를 판정한다.**
없는 것보다 나쁜 이유가 그것이다. 그래서 미측정은 미측정으로 남긴다.

## 1. 실측 완료 (4/10)

### 번들 — §F "손님 앱 JS 전송 < 350KB gzip"

| 항목 | 실측 | 예산 | 판정 |
|---|---|---|---|
| **손님 룸 first-load JS** | **205 KB** (gzip) | 350 KB | ✅ **여유 145 KB** |
| 전체 client chunk 합계 | 2.58 MB (gzip, 519 chunk) | — | 참고용 |

구성(gzip): 룸 페이지 청크 91 KB · 공유 청크 61 KB · 50 KB · webpack 2.7 KB · main-app 0.2 KB.
**단일 최대 기여자는 룸 페이지 자체(91 KB)**라, 줄일 여지를 찾는다면 거기가 먼저다.

### Tier 0 — §F "컨시어지 Tier0 응답 < 100ms (네트워크 0)"

| 측정 | p50 | p95 |
|---|---|---|
| 의도 매칭 + 가드레일 판정 | **1.0 µs** | 1.4 µs |

예산의 **1/100,000 수준**이다. Tier 0가 "네트워크 0"이라는 약속은 계산 비용 면에서
과할 만큼 지켜지고 있다 — 남은 지연은 전부 왕복(HTTP)이고, 그건 서버 측정에서 본다.

### §L L2 캐시 키 · 정원 판정 · B1 리졸버 (신규 코드의 계산 비용)

| 측정 | p50 | p95 |
|---|---|---|
| Tier1 캐시 키 계산(sha256) | 1.2 µs | 1.7 µs |
| 정원 판정 (12팀) | 0.2 µs | 0.3 µs |
| B1 통합 리졸버 (60건/주) | 0.025 ms | 0.025 ms |

**§L-D1 검증:** L2 캐시의 키 계산이 1.2µs다. 캐시 조회가 LLM 호출보다 느려질 여지는
계산 쪽엔 없고, 남는 변수는 DB 왕복 하나뿐이다(인덱스 단건 조회).

## 2. 라이브 캐시 적재 상태 (§F 외부 API 예산의 분모)

| 캐시 | 행 수 | 의미 |
|---|---|---|
| `ops_kakao_cell_index` | 4 | 다이닝 셀 4개가 warm — 그 지역은 **외부 API 0회** |
| `ops_kakao_place_cache` | 50 | 장소 50건 캐시됨 |
| `tour_translation_cache` | 70 | 번역 메모리 70건 — 이만큼은 LLM 0콜 |
| `generated_spot_content` | 0 | 아직 없음 (§L L3 프리워밍이 채울 대상) |
| `ops_concierge_cache` | 0 | 아직 없음 (§L L2, 트래픽 필요) |
| `ops_ai_usage` | 0 | 아직 없음 — **계측은 켜져 있고 트래픽이 없을 뿐이다** |

🔴 **§F "투어 1건당 LLM < 30회" / "Kakao < 20회"는 아직 판정 불가다.** 판정 수단(L0 계측)은
이제 존재하지만 실투어 트래픽이 0이다. 첫 투어가 돌면 이 쿼리가 답을 준다:

```sql
select booking_id,
       count(*) filter (where not cache_hit and outcome <> 'skipped') as billable_calls,
       count(*) filter (where cache_hit) as cache_hits
  from ops_ai_usage
 where booking_id is not null
 group by booking_id order by billable_calls desc;
```

## 3. 미측정 (6/10) — 무엇이 있어야 잴 수 있나

| §F 지표 | 예산 | 필요한 것 |
|---|---|---|
| 룸 진입 첫 유의미 렌더 | < 2.5s (4G) | dev 서버 + 실기기/스로틀 + A0.1 시뮬 데이터 |
| 메시지 전송 → 화면 반영 | < 300ms | 위와 동일 (옵티미스틱 경로) |
| 카드 탭 → 시트 열림 | < 150ms | 위와 동일 |
| 컨시어지 Tier1 응답 | < 4s | 서버 + 실 LLM 호출 (예산 소모) |
| 다이닝 카드 HIT | < 600ms, 외부 0회 | 서버 + warm 셀 |
| 다이닝 카드 MISS | **미확정** | A6.4가 먼저 실측한 뒤 예산을 쓴다(플랜 지시) |
| 라우트 p50/p95 (손님 핵심 20개) | — | 서버 + 부하 |

**착수 조건:** `npm run dev` + `ALLOW_SIM_SEED=1 npx tsx scripts/sim-tour-day.ts`.
A0.1이 그 문을 열어 뒀다 — 시뮬 예약이 집계·금전에서 배제되므로 이제 실행해도 안전하다.

## 4. 확인했고 문제없음

1. **번들이 예산 안에 있다.** 145 KB 여유 — 지금 최적화가 필요한 상태가 아니다.
2. **순수 계산 경로에 병목이 없다.** 측정한 4개 전부 마이크로초 단위다.
   느린 것이 있다면 그것은 계산이 아니라 **I/O**이고, 그건 미측정 6개 쪽에 있다.
3. **계측 인프라가 준비돼 있다.** §F의 두 예산(LLM·Kakao 호출)은 그동안 검증 불가능한
   숫자였는데, 이제 쿼리 하나로 판정된다. 트래픽만 있으면 된다.
