## 0. 🔴 이 티켓의 단일 발견: **말없는 실패**가 이 코드베이스의 지배적 결함이다

25개 중 **네 곳**에서 같은 코드를 봤다:

```ts
setState(res.ok ? 'sent' : 'idle');   // 또는 catch { setState('idle') }
```

`idle`은 **아무 일도 없던 상태**다. 즉 실패한 요청은 **아직 누르지 않은 것과 화면상 구별되지 않는다.**
성공에는 전부 확인 문구가 있는데(`sent ✓`), 실패에는 없다. 손님 입장에서 두 결과가 같아 보이므로
**손님은 성공했다고 믿는다.** 이건 UI 취향 문제가 아니라 **거짓말**이고, 이 앱에서는
"길 잃음 신고"·"분실물 신고"·"알림 켜기"처럼 **손님이 그 뒤 행동을 바꾸는** 자리에서 일어났다.

수정한 네 곳은 §1~§4. 공통 원칙으로 정리하면:

> **실패는 그 자리에 남아 있어야 하고, 다음 수단을 가리켜야 한다.**
> (채팅으로 알려주세요 / 이메일로 보내주세요 / 다시 시도)

이건 A8.3에 넘길 규율 후보다 — `useSubmitState()` 같은 공유 훅이 `idle|busy|done|failed`를
강제하면 개별 컴포넌트가 매번 기억하지 않아도 된다.

## 1. 🔴 P1 — SOS가 권한 팝업에 걸려 **영원히 출발하지 않을 수 있었다** (`SosButton`)

- **재현 경로:** 위치 권한을 아직 준 적 없는 손님이 SOS → [SOS 보내기]를 누른다.
  브라우저 권한 다이얼로그가 뜬다. 손님은 긴급 상황이라 화면만 본다(또는 팝업을 인지 못 한다).
- **관측된 동작:** `getCurrentPosition`을 **await한 뒤에** POST를 보냈다. 그런데 이 API의
  `timeout` 옵션은 **사용자가 권한 다이얼로그에서 결정하는 시간을 포함하지 않는다**(W3C 명세).
  결정하지 않으면 콜백이 오지 않고, 8초 타임아웃도 시작되지 않는다 →
  **SOS는 기기 안에 머문 채 스피너만 돈다.** 거부하면 즉시 보내지므로(주석대로),
  **거부보다 무응답이 더 나쁜** 역설적 상태였다.
- **왜 P1인가:** 이 앱에서 가장 시간이 중요한 단일 액션이다. "위치 없는 SOS"는 쓸모가 있지만
  "도착하지 않은 SOS"는 아무 쓸모가 없다.
- **조치 2가지:**
  1. **벽시계 상한 4초**(`withCap`) — 권한 대기까지 포함해서 자른다. 4초 안에 좌표가 없으면
     좌표 없이 보낸다.
  2. **좌표 요청 시점을 [SOS] 탭 시점으로 앞당김** — 동의 문구를 읽는 2~5초 동안 권한 팝업과
     GPS 획득이 **겹쳐서** 진행된다. 그래서 실제로는 대개 좌표가 붙은 채로 즉시 나간다.
     취소하면 아무것도 전송되지 않는다(동의 범위 불변).
- 회귀: `__tests__/components/tour-mode/sosLocationCap.test.tsx` — 콜백이 **영영 오지 않는**
  지오로케이션을 심고 4초 뒤 전송을 확인한다(수정 전에는 이 테스트가 타임아웃으로 실패).

## 2. 🔴 P1 — 도움 신호가 실패해도 아무 말이 없었다 (`QuickSignalBar`)

- **관측된 동작:** `if (res.ok) setSentAt(...)`. 실패하면 칩이 원래 모습으로 돌아갈 뿐이다.
  `길 잃음` · `여기로 와주세요(픽업)` · `늦어요` 가 전부 이 경로다.
- **왜 P1인가:** 실패를 모르면 **손님은 기다린다.** 길을 잃은 채로.
  코드 주석은 *"the chat composer remains the fallback"*라고 했지만,
  **폴백이 필요하다는 사실 자체가 손님에게 전달되지 않았다.**
- **조치:** `failed` 상태 + 5로케일 문구(“전달되지 않았어요 — 아래 채팅으로 알려주세요.”).
- **덤으로 잡힌 버그:** 확인 문구가 `Date.now() - sentAt < 4000`으로 **렌더 중에 계산**되고 있었다.
  이걸 되돌릴 타이머가 없어서, 조용한 룸에서는 리렌더가 없어 **"전달됐어요 ✓"가 영구히 남고
  칩이 계속 가려졌다**(= 다음 신호를 보낼 수 없다). 상태 + 타이머로 교체.

## 3. 🟡→✅ P2 — 푸시 켜기 실패가 "안 켠 것"과 똑같이 보였다 (`PushOptInBanner`)

OS 권한을 허용하고 브라우저 구독까지 만들었는데 **서버 저장이 실패하면** 배너가 `idle`로 돌아갔다.
손님은 켰다고 믿고 **집합·지연 알림을 기다리지만, 그 알림은 영원히 오지 않는다**
(구독은 브라우저에만 있고 서버는 모른다). `failed` 상태 + 5로케일 + 재시도 버튼 유지로 수정.

## 4. 🟡→✅ P2 — 분실물 신고 실패 (`EndedCard`)

같은 모양(`res.ok ? 'sent' : 'idle'`). 이 컴포넌트는 **이미 이메일 폴백을 갖고 있었는데**
(신호를 못 쓰는 손님용 `mailto`) 실패 시 그쪽으로 보내지 않았다. 실패하면 이메일 링크로 전환하고
이유를 한 줄로 말한다. 🔴 5로케일 중 **es·zh를 빠뜨린 채 커밋했더니 `tsc`가 잡았다** —
A1.5에서 칭찬한 `Record<RoomLocale, …>` 타입 강제가 실제로 작동한 사례로 기록해 둔다.

## 5. 🟡→✅ P2 — "불러오지 못함"을 "예약 없음"으로 말했다 (`TourModeEntry`)

- **재현 경로:** 투어 전날 밤, 손님이 `/tour-mode`를 연다. `/api/tour-mode/bookings`가 500을 준다.
- **관측된 동작:** 401만 구분하고 **나머지 응답은 전부 데이터로 취급**했다(`json.bookings ?? []`).
  결과: **"이 계정에 예정된 확정 투어가 없습니다."** 네트워크 예외는 한술 더 떠 `signedIn=false`로
  떨어져 "로그인하면 보입니다"라고 했다 — 이미 로그인한 손님에게.
- **왜 P2가 아니라 거의 P1인가(그래도 P2로 둔 이유):** 문구의 의미가 **"당신의 예약이 사라졌다"**여서
  투어 전날 밤에 최악의 오해를 만든다. 다만 손님에게는 이메일의 룸 링크라는 **다른 경로가 남아 있어**
  차단은 아니다.
- **조치:** `loadFailed` 상태를 `signedIn`과 **분리**하고, 5로케일 문구가 **책임 소재를 명시**한다
  (“예약 문제가 아니라 저희 쪽 문제예요”). 빈 목록의 정직한 빈 상태는 그대로 두고 테스트로 고정했다.
- 🔴 이건 B1에서 잡았던 **"비활성 ≠ 빈 상태"**와 같은 오류다. 표면만 다르다.

## 6. P3 기록 (수정 안 함)

1. **`HomeTab`·`RoomShell`의 "지금" 마커가 가시성 복귀 시 재동기화하지 않는다** — 60초 인터벌만 있다.
   같은 계열 `DepartureCountdown`은 `visibilitychange`로 재동기화한다(A1.2에서 확인).
   **고치지 않은 이유:** 브라우저는 복귀 시 밀린 인터벌을 즉시 한 번 실행하므로 실제 어긋남은
   1초 미만으로 추정된다. **재기까지 하지 않고 "아마 괜찮다"로 코드를 바꾸는 건 §L-D6 정신에 어긋난다.**
   A0.3의 실기기 측정 항목에 얹는다.
2. **`AudioButton`의 `aria-label="listen"`이 영어 고정** — A1.1이 낸 "aria-label 9개 영어 고정"과
   같은 백로그(A8.2)에 합류.
3. **`useKeyboardOpen`이 핀치 줌을 키보드로 오인할 수 있다** — `innerHeight - visualViewport.height`
   기준이라 확대 시에도 참이 될 수 있다(→ 탭바가 사라진다). **실기기 확인 전에는 추정이므로 고치지 않는다.**

## 7. 확인했고 문제없음

1. **연결 상태를 정직하게 말한다**(`RoomShell`) — `offline`/`connecting`이 5로케일로 헤더에 뜬다.
   조용히 죽지 않는다.
2. **`PlanNudgeModal`은 실패 시 뜨지 않는다** — 확인 못 한 상태로 넛지를 띄우지 않는 쪽이 옳다.
3. **`InstallBanner`가 저장소 실패를 전부 흡수한다**(스토리지 차단 브라우저에서도 배너가 뜬다).
   날짜 파싱 실패 시 **표시하지 않는** 선택도 옳다.
4. **`GuideCaptionBar`의 fire-and-forget은 정당하다** — 가이드 본인 기기의 실시간 캡션 청크이고,
   한 청크의 유실에 UI를 띄우면 발화 중 방해가 된다. (손님 신호와 성격이 다르다.)
5. **`useKeyboardOpen`이 `visualViewport` 부재 시 조용히 false** — 데스크톱에서 안전.
6. **타이머 전수 점검:** 이 25개에 조건 없는 고빈도 틱은 없다(A1.2 `NoticeBanner` 부류 재발 없음).
   `CaptionBanner`(자동 숨김)·`PushOptInBanner`(1.2초 지연 노출)는 목적이 분명하고 정리된다.
7. **`TourRoomClient`의 lint 에러는 제품 결함이 아니다** — `react-hooks/set-state-in-effect`이고
   위치는 **244행**(프로젝트 메모리에 적힌 285행은 낡았다). URL의 `?rt=` 토큰을 마운트 시 1회
   상태로 옮기는 코드라 캐스케이드가 아니다. **린터를 속이는 래핑은 하지 않았다** — 신호를 지우는 것이기 때문.
   A4.5에서 판단할 것.
