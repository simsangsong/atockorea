## 0. 한 줄 결론

> 추상적 도구 선택(**Match me / Build myself 토글**)을 없애고, 고객이 실제로 하는 결정인 **"어떤 형태의 투어인가"**를 **인원수 기반 실시간 가격·개인화 카피**와 함께 페이지 최상단에 놓는다.
> 라우팅: **Private → 큐레이션 프라이빗 리스트(빌더는 "직접 짜기" 보조)**, **그룹 → 큐레이션 상품 리스트**, **추천받기 → 매처 인페이스(묻힌 링크 아닌 동등 카드)**. 매처 코드는 보존.

## 1. 진단 — 왜 바꾸나

1. `matcher` / `builder`는 **내부 도구 taxonomy**다. 고객은 "Match me가 뭐고 Build myself가 뭔지"를 먼저 배워야 행동할 수 있다 → 전형적 마찰.
2. 정작 고객의 실제 결정(프라이빗이냐 그룹이냐, 내 인원·예산·편의에 뭐가 맞냐)을 돕는 **`ChooseTravelStyle` 카드는 5번째 섹션**(Featured + Destinations 아래)에 묻혀 있다.
3. 그 카드 셋(Small Group / Private / Bus)이 **전부 `/tours/list?type=...` 리스트로만** 가서, 타입별 차별화된 경험(프라이빗=맞춤, 그룹=둘러보기)을 못 살린다.

**결론:** 기능을 더하는 게 아니라 **정보 구조를 뒤집는다.** "도구를 쓰세요"에서 → "당신 상황엔 이게 맞고, 이유는 이것"으로.

## 2. 코드/데이터 실사 (확정 사실 — 2026-06-21 에이전트 실측)


### 2.2 `ChooseTravelStyle` 현황 (`components/home/v2/sections/choose-travel-style.tsx`)
- 카드 3장: **Small Group**(amber 추천 강조) / **Private**(car) / **Bus**(bus)
- 라우팅(현재): 전부 리스트
  - Small Group → `/tours/list?type=join`
  - Private → `/tours/list?type=private`
  - Bus → `/tours/list?type=bus`
- 가격 라벨 이미 존재: Private "$198 from /vehicle", 그룹 "from $X /person" (`lib/home/choose-style-card-usd.ts`)
- 라우팅 상수: `lib/home/home-cta-routes.ts`

### 2.3 ⚠ 데이터에 "깔끔한 3분류"가 없다 (개혁의 최대 제약)
- 가격엔진 `lib/quote-engine/pricing-policy.ts`의 `PricingTrack` = `private | cruise | dmz`. **bus / small-group 트랙 자체가 없음** — 엔진은 **per-vehicle private만** 모델링.
- 상품 데이터에 단일 `tour_type` enum **없음**. 타입은 **0~1 graded fit**(`small_group_fit` / `bus_fit` / `private_fit`)로 표현 — **배타적 분류가 아님**. 같은 상품이 `small_group_fit=1` AND `bus_fit=1`을 동시에 가짐.
- **활성 카탈로그 35개 직접 분류 (사실 정정 2026-06-21 — 사용자 지적):**

| 구분 | 개수 | 지역 분포 |
|---|---|---|
| per-vehicle 프라이빗(엔진가) | **6** | 서울 4 / 제주 1 / 부산 1 |
| **그룹 계열**(버스·소그룹·크루즈쇼어·일반 per-person day-tour) | **29** | 제주 ~12 / 부산 8 / 서울 8 |

- → **그룹 계열 리스트는 풍부**(지역당 8~12개). "그룹 → 리스트"는 빈약하지 않다. **인벤토리는 블로커 아님.**
- ⚠ **단, 배타적 라벨 카운트는 다름.** `-bus-tour` 슬러그만 = 지역당 0~1개, `-small-group` 슬러그만 = 지역당 0~1개(서울 0, 일부 `*-bus-tour`는 `lib/tour-consumer-visibility.ts` consumer-blocked). 즉 **"버스 전용 탭에 `-bus-tour`만"식 배타적 분리는 빈약**하지만, 이는 비현실적 정의. **그룹 한 바구니 또는 fit-기반 필터로 보면 전부 풍부.**
- **남는 실제 과제(인벤토리 아님):** 카드를 버스/소그룹/크루즈로 가를 때 **각 카드가 무슨 규칙으로 상품을 거를지**(슬러그 토큰 / `fit≥threshold` / 신규 필드) — 배타적 enum이 없으므로 분류 규칙을 **정의**해야 함(중복 노출 허용 여부 포함).

### 2.4 가격: 프라이빗=엔진(전 지역), 그룹=카탈로그 from-price, 크로스오버=도출
- **프라이빗 per-vehicle 가격은 엔진(`pricing-policy.ts`)이 전 지역 계산** — **제주 포함**(카탈로그 라벨은 제주 private이 "confirm at booking"으로 비어 있으니 라벨이 아니라 **엔진을 쓴다**). 언어·시간·인원 tier·지역 반영.
- **그룹 per-person 가격은 카탈로그 JSON 리터럴**(엔진 미산출). 그룹 카드의 "from" = 해당 지역 그룹 계열 **최저 per-person**(정직한 from).
- **크로스오버는 하드코딩 금지 → region별 도출:** `N* = ceil(프라이빗_엔진가 / 그룹_from_perperson)`. party ≥ N*일 때만 "✓ 프라이빗이 더 쌉니다"·프라이빗 추천 점등. 예) 부산 private 엔진가 ≈ ₩(엔진) / 그룹 from $49 → N* ≈ 8. (이전 초안의 "4명/5명 우위"는 **소그룹 $79 기준 산술의 오용** — 그룹 from(최저)으로 합치면 N*는 더 큼. 그래서 임계를 박지 않고 엔진÷from으로 계산.)
  - pax tier(엔진): **1–6 세단 / 7–9 밴(+₩50k) / 10–13 솔라티(+₩150k) / 14+ 수동** — 프라이빗가가 인원에 따라 계단 상승하므로 N*도 엔진가에 자동 반영.
- **가드:** "정확한 견적"처럼 단언 금지. 가격 옆 "다음 단계에서 확정" 한 줄. 그룹 from은 "from $X" 표기(정찰가 아님 명시).

## 3. Phase 0 게이트 — Wave 0보다 먼저 (잔여 항목 승인)

**인벤토리는 블로커 아님(§2.3 정정).** 그룹 계열 ~29개로 리스트 풍부. **카드 구조도 확정**(3카드 {프라이빗 / 그룹 / 추천받기}, 사용자 승인 2026-06-21). 따라서 게이트의 "카드 몇 장" 논쟁은 종료. 남은 건 분류 규칙 + 가격 상수 + §B 번복.

> ⚠ **게이트 순서:** 이 게이트는 **Wave 0보다 앞.** Wave 0의 U6(라우팅 IA)·U12(H1)는 카드 구조에 의존하므로, 구조 미확정 상태로 골격을 짤 수 없음. (이전 초안이 "게이트=Wave 1 직전"으로 둔 건 시퀀스 오류 — 수정함.)

**확정 카드 구조:**

| 카드 | 라우팅 | 비고 |
|---|---|---|
| **프라이빗** | → 큐레이션 프라이빗 리스트(빌더 보조)[U4] | 가격=엔진 |
| **그룹** | → 큐레이션 상품 리스트(소그룹·버스·크루즈쇼어 통합) | 타입 세분은 리스트 내 필터/배지 |
| **추천받기** | → 매처 인페이스[U3] | 동등 카드, CTA 톤만 보조 |

크루즈쇼어 독립 카드(4카드 안)는 **보류** — 우선 그룹 리스트 내 "크루즈 기항" 필터로 흡수, 트래픽 보고 승격 판단(§D 후보).

**분류 규칙 (배타적 enum 부재 → 정의 필요):**
- 그룹 리스트 = "비-프라이빗 전체" + 리스트 내 필터 칩(소그룹 / 버스 / 크루즈 기항).
- 필터 1순위: **슬러그 토큰**(`-bus-tour` / `-small-group` / `-cruise-shore`). 라벨 없는 일반 day-tour(~24개)는 **`fit≥threshold`**(예: `small_group_fit≥0.8`)로 소그룹 필터에 포함. (중복 노출 허용)
- `/tours/list?type=…` 라우팅 재사용(이미 존재). `lib/tour-consumer-visibility.ts` 블록리스트 존중.

**Phase 0 잔여 산출물(승인 대기):** (a) 분류 규칙 threshold 서명, (b) 가격 상수·N* 도출식 리뷰(§2.4), (c) §B 번복 로깅 승인, (d) 서울 destination 상태 확정(§4). → 통과 전 **Wave 0** 착수 금지(go/no-go).

## 3.5 채택한 13 업그레이드 (CRO/UX 리뷰 2026-06-21, 사용자 "13개 전부 채택")

세계 최고 CRO/UX 관점 리뷰에서 도출, **13개 전부 채택**. 단 **한 방에 다 만들지 않고 0→1→2 웨이브로 순차 실행**(§7). 13개는 잡탕이 아니라 **인원수(party)를 척추로 묶인 하나의 개인화 시스템** — U1(실시간 가격)·U5(동적 배지)·U9(운반)·U10(칩)이 모두 같은 party 입력을 공유.

|---|---|---|---|
| **U1** | **인원수 그대로 넣은 실시간 가격 비교**(그룹 GP×party vs 프라이빗 엔진 PV·1인당 PV/party) — 말하지 말고 그들 숫자로 증명. 가격 출처/임계는 §2.4·§5 | 🔴 최고 | 1 |
| **U2** | **인원수 관문 금지** — 기본 2인 프리필 + 카드 즉시 노출(Wave 0). 조정 시 가격 실시간 갱신의 payoff는 U1과 함께 Wave 1에 도착 | 🔴 | 0 |

## 4. 목표 정보 구조 (제안 흐름 — 13 업그레이드 반영)

```
1. Hero (사진 + H1[U12 리프레이밍])              ← 화보 톤 결정들 보존
2. 목적지(Jeju / Busan / Seoul*) + 인원 스테퍼[U6·U2] ← 기본 2인 프리필, 관문 아님. destination이 가격/재고 종속
3. 투어타입 3카드 (인원 실시간 개인화)[U1·U5]
       ├─ 프라이빗 → 큐레이션 프라이빗 리스트[U4] (빌더는 "직접 짜기" 보조)
       ├─ 그룹    → 큐레이션 상품 리스트 (소그룹·버스·크루즈 통합, 리스트 내 필터)
       └─ 추천받기 → 매처 인페이스 결과[U3]        ← 동등 카드(묻힌 링크 아님)
   ─ 카드 묶음 바로 아래: 리스크 리버설 1줄[U7] (카드마다 반복 금지 — 1회만)
4. (이하 Destinations / Why / Process / FinalCTA 유지)
   전 구간: party·date·lang 끝까지 운반[U9], 퍼널 계측[U13]
```
*서울 destination = **노출 + 투어 재활성화 확정**(사용자 결정, §11.4). 중단 hold를 되돌려 실inventory로 노출. 재활성화는 Wave 0 선행 step.

**카드 콘텐츠 예산 (简单粗暴 규율 — 카드당 최대 3요소):**
1. 핵심 1줄, 2. 실시간 가격 1줄[U1], 3. 동적 추천 배지[U5]. → **여기까지가 카드 표면.**
- Best-for 태그·정원·장단점 = 펼침(progressive disclosure)로 내림.
- 리스크 리버설[U7]·긴급성[U8] = 카드가 아니라 **카드 묶음 아래 섹션 1회.**

설계 원칙:
- **인원수는 관문이 아니라 기본값+실시간**[U2] — 카드 즉시 보이고, 조정하면(payoff는 Wave 1 U1) 가격·추천 배지가 변함. 닭-달걀은 "기본 2인"으로 해소.
- 매처는 **묻지 않되 동등 카드**[U3]로 — 미결정 세그먼트 보존.

## 5. 카드 카피 (검증상수 기반, 6 로케일)

개인화 변수: `{party}` = 인원(기본 2), `PV` = 프라이빗 엔진 per-vehicle 가격(§2.4, region·party 반영), `GP` = 그룹 from per-person(§2.4), `N* = ceil(PV / GP)` = 프라이빗이 그룹보다 싸지는 인원.

- **프라이빗 (→큐레이션 프라이빗 리스트, 빌더 보조)**[U4]
  - 핵심: "내 일정·픽업·페이스 그대로, 차 한 대 통째로."
  - **실시간 가격**[U1]: "{party}명 → 1대 **PV** (= 1인당 **PV/{party}**)" + `{party} ≥ N*`일 때만 "✓ 그룹보다 쌉니다"
  - 추천 배지[U5]: `{party} ≥ N*`에서 점등 (가격이 실제로 역전될 때만 — 정직)
  - 펼침: 호텔 픽업 / 프라이버시 / 정원(세단 6·밴 13)
- **그룹 (→큐레이션 상품 리스트)**
  - 핵심: "검증된 코스를 가성비로."
  - **실시간 가격**[U1]: "1인당 **GP**부터 → {party}명 ≈ **GP×{party}**" (카탈로그 from)
  - 추천 배지[U5]: `{party} < N*`에서 점등
  - 펼침: 소그룹/버스/크루즈 필터 안내 / 출발 확정
- **추천받기 (→매처 인페이스)**[U3]
  - 핵심: "어떤 게 맞을지 모르겠다면 — 30초면 추천해 드려요."
  - 동등 카드. CTA 톤만 보조(아웃라인). 미결정 세그먼트 흡수.

**가격 출처 가드(§2.4):** 프라이빗 PV = **엔진**(전 지역, 제주 포함), 그룹 GP = **카탈로그 from-price**. 임계 N*는 **하드코딩 금지 → `ceil(PV/GP)` 도출.** "✓ 쌉니다"·프라이빗 추천은 **실제 역전(party≥N*)에서만** — 4명에 그룹이 더 싼데 프라이빗을 "이득"이라 하면 허위. "정확 견적은 다음 단계 확정" 한 줄 병기.
**카피 가드(§B):** "최적/Best match" 류 과약속 금지.

## 12. 비주얼 디자인 — 고급(premium) + 직관(intuitive)

**전제:** 기존 `ChooseTravelStyle` 비주얼은 이미 정제됨(다크 슬레이트+앰버 / 화이트 민트 팟 + 레이어드 inset·soft 섀도우, `rounded-card`·`shadow-2`·scroll-reveal·`SnapScrollDots`·hover `-translate-y-0.5`·motion-reduce 가드·모바일 snap). **새로 만들지 않고 상속·확장**한다. 신규 라이브러리 0(framer만).

### 고급스러움 (premium) — 5 레버
- **V1. 에디토리얼 타이포** — 섹션 타이틀/eyebrow를 Destinations와 같은 매거진 세리프(Noto Serif KR → Cormorant, 이미 layout 로드됨). 카드에 작은 세리프 eyebrow("Private day"/"Group tour"). italic 금지(§B 2026-05-20).
- **V2. 사진 텍스처(선택, 임팩트 큼)** — 카드 상단 얇은 듀오톤 사진 스트립(프라이빗=해안/프라이빗카 / 그룹=소그룹 순간) + 기존 B2/Vogue 필터(`saturate/contrast/brightness`) + 필름그레인. 히어로↔Destinations↔카드 이미지 아이덴티티 통일. (WebP, LCP-safe: fold 아래/lazy)

### 직관성 (intuitive) — 6 레버
- **V6. 인원 스테퍼 = 명백한 컨트롤** — 카드 바로 위 슬림 pill 스테퍼(− 2 +), 탭타깃 ≥44px + "{party}명 기준" 캡션. 바꾸면 카드 눈앞 갱신 = 즉각 인과(핵심 직관 장치).

- **V9. 경로 의미를 위계로** — 프라이빗/그룹 = solid primary CTA(`homeBtnPrimary`), 추천받기 = **ghost/outline**(도우미). 무게로 메인 vs 안전망 구분.
- **V10. 점진적 공개** — "자세히 ▾"로 정원(세단 6/밴 13)·장단점 노출. 표면 calm 유지.
- **V11. 연속성 어포던스** — 고른 목적지+인원이 다음 화면으로 보이게 따라감(작은 요약 칩, U9와 연계). "기억했나?" 불안 제거.
